Problem

AssetPanda's knowledge centre (help.assetpanda.com) was a flat, informally written HTML site that had evolved organically without content governance, taxonomy, or structural planning. Users couldn't find what they needed and filed repeat support tickets. The content was not structured for AI retrieval, making bot-assisted support impossible.

Solution

Conducted a full content audit, designed a topic-based information architecture from scratch, initially migrated content to ProProfs as a quick-win solution under infrastructure and budget constraints, then evolved the platform to GitBook (prohelp.assetpanda.com) for enterprise-grade structure. Established style governance across both phases and trained AssetPanda's AI assistant (Ursa AI) on the final documentation corpus for RAG-based retrieval.

Phase 1

Content Audit and Gap Analysis

I started by inventorying every page on help.assetpanda.com. Each article was catalogued, assessed for accuracy against the current product, and tagged by topic area. Simultaneously, I analysed support ticket patterns to identify recurring questions that lacked corresponding documentation.

This dual approach (existing content audit + support data analysis) produced a prioritised gap report that became the roadmap for the migration.

🚩 Red Flag Informal tone throughout. Example from the Getting Started page: "So you've got this great, new Asset Panda application. Exciting, isn't it? If you're like us, you just can't wait to unwrap this new toy and see it in action." This reads as marketing copy, not user documentation.
🚩 Red Flag Flat file structure with no hierarchy. Pages like Integrations.html, Support_Menu.html, and Asset_Panda_Navigation.html exist as isolated HTML files with no parent-child relationships or categorical grouping.
🚩 Red Flag Navigation relies entirely on in-text hyperlinks. There is no sidebar, no breadcrumb trail, no site map, and no search. Users who land on a deep page have no way to orient themselves.
🚩 Red Flag Integration documentation is minimal. The Integrations page says "integrations!" with a drum roll emoji, then provides a generic two-step instruction. No SSO guides, no prerequisite frameworks, no data flow explanations.

Reframing Problems as User Questions

To ensure the new IA was user-centred, I reframed the audit findings as questions real users would ask. This shifted the focus from "what content exists" to "what do users need to accomplish."

Audit FindingUser Question It Reveals
#1 Users can't find SSO setup instructions How do I connect my company's identity provider to AssetPanda?
#2 Support tickets repeat for basic setup tasks What are the first things I need to configure after signing up?
#3 Mobile app features are undocumented What can I do in the mobile app that I can't do on the web?
#4 No release notes exist What changed in the latest update and how does it affect my workflow?
#5 Integration docs lack decision guidance Which integration is right for my use case, and what do I need before I start?

Information Architecture Design

Phase 3

Style Guide and Content Templates

I established a documentation style guide and a standardised article template to ensure consistency at scale. Every article followed the same structure:

Template Structure Goal-oriented title (action verb + user outcome) → Brief introduction (what, who, why) → Prerequisites → Step-by-step procedure with UI navigation paths → Embedded video walkthrough (where available, repurposed from product adoption training) → Expected outcome → Troubleshooting (2 to 3 common issues) → Related articles
Voice and Tone Rules Second-person ("You can configure..."), professional but approachable, present tense for instructions, consistent action verbs: "Click" for buttons, "Select" for dropdowns, "Enter" for text fields, "Toggle" for switches
Visual Content Strategy No static screenshots. Instead, embedded product adoption training videos within relevant articles for visual guidance. Written procedures use explicit UI navigation paths (e.g., "Go to Settings > Preferences > Ursa AI") so content stays accurate across product updates without requiring image maintenance.
Content Governance Model Controlled-access publishing so product managers can update content autonomously. Version tracking aligned with sprint release cycles. "Last updated" timestamps on every article to signal content freshness. Prerequisite chains linking dependent articles through navigation (Previous/Next) and in-text cross-references.
Phase 4

Phased Migration and Content Production

The migration happened in two deliberate stages, driven by real-world constraints.

Stage 1: ProProfs (Quick Win)

When I joined, there was no dedicated documentation infrastructure and limited budget for tooling. I chose ProProfs as the initial platform because it was low-cost, required no engineering support to set up, and gave us an immediate publishing capability. This allowed me to start migrating and rewriting legacy content right away rather than waiting for infrastructure decisions to be finalised. During this stage, I established the content templates, style guide, and began the systematic rewrite of all legacy articles.

To accelerate content coverage, I repurposed video recordings from product adoption training sessions that had been delivered directly to customers. These walkthroughs were embedded within relevant KB articles, giving users both written step-by-step instructions and a visual demonstration of the same workflow. This approach maximised existing assets without additional production effort, bridging documentation gaps while the written content was being built out.

💡 Why This Mattered Starting with ProProfs meant the documentation improvement was visible to stakeholders within weeks, not months. It built confidence in the documentation programme and created momentum for the eventual platform upgrade.
💡 Resourcefulness Reusing product adoption training videos as embedded KB content meant users got visual guidance immediately, without waiting for purpose-built tutorials. It also validated which workflows users struggled with most, informing the written documentation priority.
⚡ Constraint ProProfs served the immediate need but lacked the structured navigation, version control, search sophistication, and controlled-access publishing that a growing B2B SaaS product required at enterprise scale.

Stage 2: GitBook (Enterprise Scale)

Once the content strategy was proven and the documentation programme had stakeholder buy-in, I migrated the entire corpus to GitBook (prohelp.assetpanda.com). GitBook provided the capabilities that ProProfs could not: hierarchical left-nav with collapsible sections, full-text search with keyboard shortcuts, version control, controlled-access publishing for PM self-service updates, and a content structure that supported AI retrieval.

All content was restructured into the final IA during this migration. Net-new content was authored for previously undocumented features. Integration documentation (Okta SSO, Microsoft Entra ID, Google Workspace, Lenovo Warranty, and others) was authored from scratch with prerequisite frameworks, decision trees, and governance checklists.

💡 Key Decision I designed a controlled-access publishing model on GitBook that enabled product managers to update documentation autonomously, reducing the documentation turnaround dependency on the writing team.
💡 Key Decision Content was version-controlled and aligned with sprint release cycles. Documentation updates shipped with every product release, not as an afterthought weeks later.
Phase 5

AI Enablement: Training Ursa AI

The structured knowledge base became the training corpus for AssetPanda's built-in AI assistant, Ursa AI. But documenting an AI feature required a fundamentally different content architecture from standard KB topics. I designed the Ursa documentation as a dedicated use case section with its own IA, distinct from the feature-centric structure used across the rest of the knowledge base.

Two Architectures, One Knowledge Base

The Ursa use case section at prohelp.assetpanda.com/use-cases/ursa follows a different structural pattern from standard KB topics like "Setting Up the Organization." Here is why, and how.

DimensionStandard KB TopicsUrsa Use Case Section
Organisation principle Feature-centric. Organised by platform modules: Logging In, Settings, Collections, Forms, Automations, Reporting. Capability-centric. Organised by what the AI can do for you: Answer questions, Build automations, Generate filters, Create collections.
Audience segmentation Single audience per section (typically admins or all users). Multi-audience within one section: AI Overview (all users), Administrator Guide (admins managing Ursa), feature-specific guides (Chat, Filtering, Account Builder, File Upload) for task-specific users.
Content pattern Procedural: "Go to Settings > Select X > Click Y." Step-by-step with explicit UI navigation paths and embedded training videos. Conversational + Procedural hybrid. Includes natural language prompt examples ("Show me all laptops assigned to Denver"), expected AI responses, and tips for crafting effective prompts alongside traditional step-by-step instructions.
Nesting depth Typically one level: Category > Article. Two to three levels: Ursa > Overview > FAQs; Ursa > Admin Guide > Automations, Forms. Deeper nesting reflects the AI feature's multiple interaction modes.
Governance content Not applicable. Standard features don't require usage monitoring documentation. Dedicated Admin Guide covering: enable/disable controls, AI chat log review, usage monitoring, feedback rating analysis, and quality assurance patterns. AI-specific governance that doesn't exist elsewhere in the KB.
FAQ inclusion No dedicated FAQ sub-pages for individual features. Dedicated Ursa FAQs sub-page. AI features generate unique user questions about accuracy, trust, data access, and limitations that don't arise with deterministic features.
Interaction modes Not applicable. Standard features have a single behaviour model. Three distinct modes documented on a dedicated Chat Modes page: Ask Mode (read-only, information retrieval), Agent Mode (read-write, can create forms/automations/filters and process files), and Account Builder Mode (context-restricted, setup-only). Each mode has its own prompt examples, capability boundaries, and a comparison table. This is closer to documenting a product within a product.
Example content UI navigation paths and embedded training videos showing feature workflows. Prompt-and-response examples showing what users type and what Ursa returns. Includes suggested prompts for different use cases, teaching users how to interact with an AI (a form of prompt engineering for end users).

Key Architectural Decisions for AI Documentation

Capability-first, not feature-first Standard KB topics are organised by platform module (Settings, Collections, Automations). The Ursa section is organised by what users can accomplish with AI: answer questions, build automations via natural language, generate filters, create collections with suggested fields. This matches how users think about AI ("What can it do for me?") rather than how the platform is built internally.
Prompt examples as a content pattern Regular KB articles use explicit UI navigation paths and embedded training videos to guide users through workflows. Ursa articles introduce a completely different content pattern: natural language prompt examples and expected responses ("Show me all laptops assigned to Denver"). This is effectively end-user prompt engineering guidance, a content type that doesn't exist anywhere else in the KB. It teaches users how to converse with AI, not just how to navigate menus.
AI governance as a first-class topic No standard KB feature needs a dedicated admin governance page. Ursa's Administrator Guide covers enable/disable controls, AI chat log review, usage monitoring, and feedback rating analysis. This content type exists because AI features require oversight that deterministic features do not. I created this as a distinct article so admins could find governance controls without wading through end-user content.
Single-capability pages for RAG precision Each Ursa sub-page (Chat, Filtering, Account Builder, File Upload) covers exactly one capability. This was a deliberate choice for AI retrieval: when Ursa itself needs to answer a question about one of its own capabilities, the retrieval system can pull a precise, scoped article rather than a chunk from a multi-topic page. The documentation serves both human readers AND the AI that reads it.

Documenting AI Interaction Modes: A New Content Challenge

Ursa operates in distinct modes, each with different capabilities, constraints, and documentation needs. This is something that has no parallel in traditional feature documentation. A standard platform feature like "Automations" has one behaviour. Ursa's Chat Modes page documents three separate interaction paradigms within a single feature.

ModeBehaviourDocumentation Challenge
Ask Mode
(Default, read-only)		 Information retrieval only. Ursa answers questions and looks up data but cannot modify anything. Fastest response times, no file upload support. Had to clearly communicate the safety boundary: users can explore freely without risk of changing data. Content uses "read-only" framing to set expectations and reduce anxiety for new AI users.
Agent Mode
(Advanced, read-write)		 Full action capability. Ursa can create forms, build automations, apply filters, and process uploaded files. Uses specialised tools behind the scenes. Needed to document both capability and caution. Users must understand what Agent Mode can do AND that they should review outputs before saving. Content includes "review before activating" guardrails that don't exist for Ask Mode.
Account Builder Mode
(Specialised, setup-only)		 Available only on the Account Builder page. Generates a complete account hierarchy (collections, fields, categories) from a business description. Context-restricted mode needed its own content scope. Unlike Ask and Agent which are available everywhere, this mode only appears in a specific UI context. Documentation had to prevent users from searching for it in the wrong place.
Why this required a different content architecture Standard KB features don't have "modes." A feature either does something or it doesn't. Ursa's modal design meant I had to create a content structure that explains when and why a user should switch modes, with a comparison table showing capability differences across all three modes. This is closer to documenting a product within a product than documenting a single feature.
Prompt examples per mode Each mode needed its own set of example prompts to teach users the right interaction pattern. Ask Mode examples are questions ("How many records...?"). Agent Mode examples are instructions ("Create a form for..."). Account Builder examples are descriptions ("Set up a fleet tracking account"). The prompt style changes with the mode, and the documentation reflects that.
Mode switching as a documented workflow I documented the mode-switching interaction itself as a procedure (open chat drawer, find mode dropdown, select mode) and included a practical tip: "Start with Ask Mode for quick questions. Switch to Agent Mode when you need Ursa to create something or process a file." This kind of usage guidance doesn't exist in standard feature documentation because standard features don't have interaction modes.

Content Structured for Dual Consumption

The Ursa documentation serves two consumers simultaneously. Human users read it to learn how to use the AI assistant. Ursa AI itself retrieves from this same corpus when users ask questions about its capabilities. Every structural decision (single-topic pages, consistent terminology, clear scope boundaries, logical hierarchy) was made to optimise for both consumption paths.

Chunking for retrieval One topic per article. Each section is self-contained and can stand alone as a meaningful retrieval result without requiring context from other sections.
Hierarchy for context The category and sub-section structure (e.g., Use Cases > Ursa > Ursa Chat > Chat Modes) provides implicit context for retrieval. The AI can infer scope from the article's position in the hierarchy without needing explicit tags.
Terminology for matching All product terms standardised to exact UI labels. No ambiguous synonyms. The AI matches user queries against consistent terminology.
Escalation for safety Defined which queries Ursa AI should answer vs route to human support. Account-specific issues, billing, bug reports, and security incidents always escalate.
