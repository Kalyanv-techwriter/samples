Actifio VDP Data Protection Lifecycle

A business-process-driven guide to the Virtual Data Pipeline RESTful API

The Actifio Virtual Data Pipeline (VDP) RESTful API enables programmatic control over the complete data protection lifecycle. Rather than presenting endpoints as isolated references, this guide follows the business workflow that enterprise teams execute when protecting, managing, and accessing their data through VDP.

VDP virtualises enterprise data by capturing a single golden copy from production and maintaining incremental-forever snapshots. Virtual read/write copies can then be instantly mounted for any purpose: disaster recovery, test/dev provisioning, analytics, or compliance reporting.

Who is this guide for? System administrators, DevOps engineers, and integration developers who need to automate data protection workflows, integrate VDP with CI/CD pipelines, or build custom orchestration around Actifio's copy data management capabilities.

1 Authenticate

Phase 1: Session Management

Business Context

Before any data protection operation can be performed, the calling application must establish an authenticated session with the Actifio appliance. VDP uses session-based authentication where a session ID acts as a temporary access token for all subsequent API calls. This session model ensures that each automation run operates within a defined security context with appropriate role-based permissions.

The following data flow describes the authentication sequence:

  • Client sends credentials via HTTP Basic Authentication to the session endpoint. The Actifio appliance validates the credentials against its local user store or connected directory service (LDAP/AD).
  • Appliance returns a session ID (UUID format) along with the session timeout value. This session ID must be included in the Authorization header of every subsequent request.
  • Client performs API operations using the session ID. Each request resets the inactivity timer.
  • Client terminates the session explicitly via the DELETE endpoint when operations are complete. This frees server resources immediately rather than waiting for timeout.
Best Practice: Always wrap your API automation in a try/finally block that deletes the session on completion, even if an error occurs. Orphaned sessions consume server resources until they time out (default: 30 minutes).
POST /actifio/session

Authenticates with the Actifio appliance and returns a session ID for subsequent API calls.

Authentication

HTTP Basic Authentication. Pass credentials as username:password in the Authorization header.

Sample Request

# Authenticate and capture the session ID curl -X POST \ --user admin:password \ -H "Content-Type: application/json" \ -k https://10.0.0.1/actifio/session

Response Fields

FieldTypeDescription
idstringSession ID in UUID format. Include this in the Authorization header as Actifio {id} for all subsequent requests.
usernamestringAuthenticated username for this session.
timeoutintegerSession inactivity timeout in seconds. Default: 1800 (30 minutes).

Sample Response (200 OK)

{ "id": "61eaf9e3-eca3-487b-a62a-9d7d2931a84d", "username": "admin", "timeout": 1800 }
DELETE /actifio/session/{session_id}

Terminates an active session and releases server resources. Always call this endpoint after completing API operations.

Path Parameters

ParameterTypeDescription
session_idstringThe session ID returned by the POST /session endpoint.

Sample Request

curl -X DELETE \ -k https://10.0.0.1/actifio/session/61eaf9e3-eca3-487b-a62a-9d7d2931a84d

Response: 204 No Content (empty body on success).

2 Discover

Phase 2: Application Discovery

Business Context

Before you can protect data, you need to know what data exists. The Discovery phase retrieves the inventory of applications (databases, VMs, file systems) that the Actifio appliance has discovered on connected hosts. Each application represents a workload that can be protected by VDP.

In a typical enterprise workflow, discovery happens in two stages:

  • List all applications to get a broad inventory of what the appliance can see. Filter by application type (VMBackup, SQLServerWriter, OracleDB, FileSystem) to focus on specific workload categories.
  • Inspect individual applications to retrieve detailed metadata including host information, current SLA assignment, protection status, and last backup timestamp. This determines which applications need protection policies applied.
Figure 2: Application Discovery and Assessment Flow
GET /actifio/application

Retrieves the inventory of applications discovered by the Actifio appliance. Applications represent workloads (databases, VMs, file systems) that can be protected by VDP.

Query Parameters

ParameterTypeRequiredDescription
filterstringNoFilter expression. Format: field:==value. Example: apptype:==VMBackup
limitintegerNoMaximum results. Default: 50. Maximum: 500.
offsetintegerNoRecords to skip for pagination. Default: 0.
sortstringNoSort field and order. Format: field:asc or field:desc.

Sample Request

# List all SQL Server applications curl -X GET \ -H "Content-Type: application/json" \ -H "Authorization: Actifio 61eaf9e3-eca3-487b-a62a-9d7d2931a84d" \ -k "https://10.0.0.1/actifio/application?filter=apptype:==SQLServerWriter&limit=10"

Response Fields

FieldTypeDescription
idstringUnique application identifier.
appnamestringDisplay name (database name, VM name, etc.).
apptypestringValues: VMBackup, SQLServerWriter, OracleDB, FileSystem, SystemState.
hostnamestringHost on which the application resides.
slaobjectAssigned SLA policy. Contains id and name.
protectionstatusstringValues: protected, unprotected, partial.
lastbackuptimestringISO 8601 timestamp of last successful backup.

Sample Response (200 OK)

{ "count": 2, "items": [ { "id": "40215", "appname": "proddb_sqlserver", "apptype": "SQLServerWriter", "hostname": "sql-prod-01.corp.local", "sla": { "id": "5021", "name": "Gold-4Hr-RPO" }, "protectionstatus": "protected", "lastbackuptime": "2021-11-15T04:30:00Z" } ] }
Automation Use Case: A common DevOps pattern is to run a nightly script that calls GET /application with filter=protectionstatus:==unprotected, then automatically assigns an SLA policy to any newly discovered workloads. This ensures no application goes unprotected after deployment.
3 Protect

Phase 3: Backup Orchestration

Business Context

Protection in VDP is policy-driven. Service Level Agreements (SLAs) define the backup frequency, retention period, and replication targets for each application. Once an SLA is applied to an application, VDP automatically executes backups according to the schedule. However, the API also supports on-demand backups for scenarios like pre-deployment snapshots, pre-migration captures, or compliance-triggered archival.

The protection workflow involves three steps:

  • List available SLA templates to understand what protection policies exist. Each SLA defines the RPO (Recovery Point Objective), retention window, and storage tier targets.
  • Apply an SLA to an application to activate scheduled protection. The appliance begins executing backups according to the SLA schedule.
  • Trigger on-demand backups when immediate protection is needed outside the scheduled window.
POST /actifio/backup

Triggers an on-demand backup for a specific application. Use this when you need an immediate point-in-time capture outside the SLA schedule.

Request Body

FieldTypeRequiredDescription
appidstringYesApplication ID from the Discovery phase.
policyidstringYesSLA policy ID. Determines backup type and retention.
labelstringNoCustom label for this backup (e.g., "pre-deploy-v2.4").

Sample Request

# Trigger on-demand backup before deployment curl -X POST \ -H "Content-Type: application/json" \ -H "Authorization: Actifio 61eaf9e3-eca3-487b-a62a-9d7d2931a84d" \ -d '{ "appid": "40215", "policyid": "5021", "label": "pre-deploy-v2.4" }' \ -k https://10.0.0.1/actifio/backup

Sample Response (200 OK)

{ "id": "92145", "jobname": "Job_0092145", "status": "running", "progress": 0, "appname": "proddb_sqlserver" }
Important: The POST /backup endpoint returns immediately with the job ID and a running status. The backup executes asynchronously. Use the Phase 5 monitoring endpoint (GET /jobstatus) to track progress and confirm completion before proceeding with deployment.
4 Access

Phase 4: Data Virtualisation and Instant Mount

Business Context

This is where VDP's core value proposition comes to life. Traditional data recovery requires copying entire datasets back to production, which can take hours for large databases. VDP's instant mount technology presents a virtual read/write copy of backup data directly to a target host in minutes, regardless of dataset size. This enables:

  • Test/Dev provisioning: Mount a production database copy to a development server for testing without moving terabytes of data.
  • Disaster recovery: Instantly mount a backup image to a recovery host to restore service within minutes of an outage.
  • Analytics and reporting: Mount a point-in-time copy to an analytics cluster without impacting production performance.
  • Compliance and forensics: Access historical data from any retention point for audit or investigation purposes.
Figure 3: Instant Mount Workflow for Test/Dev Provisioning
POST /actifio/backup/{backup_id}/mount

Creates a virtual read/write mount of a backup image on a target host. The mount is available within minutes regardless of the dataset size because VDP presents the data in-place from the snapshot pool without copying.

Path Parameters

ParameterTypeDescription
backup_idstringBackup image ID from the GET /backup response.

Request Body

FieldTypeRequiredDescription
hoststringYesTarget host name or ID where the image will be mounted.
mountpointstringNoCustom mount path on the target host. If omitted, VDP assigns a default path.
labelstringNoCustom label for this mount (e.g., "qa-refresh-sprint-42").

Sample Request

# Mount a database backup to the QA server curl -X POST \ -H "Content-Type: application/json" \ -H "Authorization: Actifio 61eaf9e3-eca3-487b-a62a-9d7d2931a84d" \ -d '{ "host": "sql-qa-01.corp.local", "label": "qa-refresh-sprint-42" }' \ -k https://10.0.0.1/actifio/backup/85082/mount

Sample Response (200 OK)

{ "id": "92301", "jobname": "Job_0092301", "status": "running", "mounthost": "sql-qa-01.corp.local", "sourcebackup": "85082" }
5 Monitor

Phase 5: Job Monitoring

Business Context

Backup and mount operations execute asynchronously. The monitoring phase provides visibility into job progress, status, and outcome. In automated pipelines, this endpoint is polled at intervals to determine when an operation has completed before proceeding to the next step (e.g., waiting for a backup to finish before starting a deployment, or waiting for a mount to complete before running test suites).

GET /actifio/jobstatus?filter=jobname:=={job_name}

Retrieves the current status and progress of a running or completed job.

Response Fields

FieldTypeDescription
jobnamestringSystem-generated job identifier.
statusstringValues: running, succeeded, failed, cancelled.
progressintegerPercentage complete (0 to 100).
startdatestringISO 8601 timestamp when the job started.
durationstringElapsed time in human-readable format.
errorcodeintegerError code if the job failed. 0 for success.
errormessagestringDescriptive error message if applicable.

Polling Pattern

# Poll every 30 seconds until job completes JOB_NAME="Job_0092301" while true; do STATUS=$(curl -sS -X GET \ -H "Authorization: Actifio $SESSION_ID" \ -k "https://10.0.0.1/actifio/jobstatus?filter=jobname:==$JOB_NAME" \ | jq -r '.items[0].status') echo "Status: $STATUS" if [ "$STATUS" = "succeeded" ] || [ "$STATUS" = "failed" ]; then break fi sleep 30 done

API Integration Best Practices

Session Management

Always delete sessions explicitly after completing operations. Wrap your automation in try/finally blocks. Never store session IDs in logs or pass them as URL parameters.

Filtering and Pagination

Use filter parameters to limit result sets. Large unfiltered queries impact appliance performance. Implement pagination with limit and offset for datasets larger than 50 records.

Asynchronous Operations

Backup and mount operations return immediately with a job ID. Always poll GET /jobstatus to confirm completion before proceeding. Recommended polling interval: 30 seconds for backups, 15 seconds for mounts.

Error Handling

Implement retry logic with exponential backoff for 503 errors. For 401 errors, automatically re-authenticate and retry the failed request once. Log all 500 errors with the full response body for support escalation.

Security

Use HTTPS for all API communication. The -k flag in curl examples bypasses certificate verification for testing only. Production environments must use valid TLS certificates.

CI/CD Integration Pattern: Embed VDP API calls in your deployment pipeline: trigger a pre-deploy backup (Phase 3), wait for completion (Phase 5), proceed with deployment, then on failure, mount the pre-deploy image (Phase 4) to the production host for instant rollback.
